以下是深度研究后的实现路径与技术原理剖析（无代码，仅供理论与逻辑参考）：

### 1. 核心技术基础：LuaTeX 的 `pdfe` 库
LuaTeX 引擎内置了一个专门处理 PDF 的底层库 —— **`pdfe` (PDF Extraction 库)**。这使得 LuaLaTeX 拥有了在编译期间“拆解”外部 PDF 的能力，而不仅是插入它。
你可以通过 Lua 脚本在插入该 PDF 之前，调用 `pdfe.open("xxx.pdf")` 打开该文件，直接读取该 PDF 的内部字典（Dictionary），以此实现书签数据的无损提取。

### 2. 需求一：将子书签提取并放置在 LaTeX 对应的层级下
* **书签树的遍历**：PDF 的标准结构中，书签存储在 `Catalog` (根目录) 下的 `/Outlines` 字典里。它是一个以 `/First` 和 `/Next` 构成的链表树。你可以用 Lua 遍历这棵树，提取出每个书签的名称 (`/Title`) 和目标位置 (`/Dest` 或 `/A`)。
* **层级对接**：在遍历时，你可以记录当前遍历的“深度（Depth）”。LaTeX 生成的 `\chapter` 通常是 Level 0，`\section` 是 Level 1。你只需要给提取到的 PDF 书签加上一个 **Base Offset（基础偏移量）**。
* **生成书签体系**：借助 LaTeX 的 `bookmark` 宏包（比默认的 `hyperref` 更加强大和灵活），可以在 Lua 脚本中动态输出类似以下格式的代码：
  `\bookmark[level=2, dest={...}]{子书签名称}`
  这样，原本位于子 PDF 顶层的书签（Level 0），就会完美挂载到你主文档的 `\section`（Level 1）下方，成为 Level 2 的书签。

### 3. 需求二：控制书签显示的默认展开层级
这也是完全可以做到的，且有两种维度的控制方式：
* **方式 A：完全继承原 PDF 的展开状态（原生逻辑）**
  在 PDF 标准规范中，每个包含子书签的节点都有一个 `/Count` 属性。
  * 如果 `/Count` 是**正数**，表示该节点在 PDF 中**默认是展开**的。
  * 如果 `/Count` 是**负数**，表示该节点在 PDF 中**默认是折叠**的。
  用 `pdfe` 库读取到这个 `/Count` 值后，如果是负数，你可以在输出时给 `\bookmark` 加上 `open=false` 参数；如果是正数则加上 `open=true`。这样可以 **100% 还原原 PDF 作者设定的展开状态**。
* **方式 B：统一使用主 LaTeX 的展开逻辑**
  如果你希望整个合并后的书签树遵循 LaTeX 代码的全局展开规则（例如只展开到 `\section`），你可以直接使用 `bookmark` 宏包提供的 `bookmarksopenlevel` 设定。在生成的书签代码中忽略原文件的 `/Count`，交由主文档的 `hyperref` / `bookmark` 引擎接管。

### 4. 需要注意的技术暗礁（坑点）
既然你需要深度研究，在后续实际落地时，有三个核心难点需要通过逻辑来处理：

1. **页面映射 (Destination Mapping)**
   原 PDF 的书签跳转目标并不是简单写着“第 5 页”，而是指向一个**对象 ID (Object ID)**。你的 Lua 脚本需要顺藤摸瓜，在子 PDF 的 Page Tree 中查找这个 ID 对应的是该文件的第几页（比如第 5 页）。然后配合 `\includepdf[link=true]{file.pdf}`，将书签目标链接到 `file.pdf.5` 这个自动生成的锚点上。
2. **字符编码 (Encoding)**
   由于历史原因，PDF 内部的书签字符串 (`/Title`) 通常是以 `PDFDocEncoding` 或者是带 BOM 的 `UTF-16BE` (以 `FE FF` 开头) 存储的。提取出来后，必须在 Lua 中进行一次解码转换，变成标准的 `UTF-8` 字符串再喂给 LaTeX，否则生成的书签会是乱码。
3. **动作与目标 (Actions vs Destinations)**
   PDF 书签的跳转有时存在 `/Dest` 字段里，有时包裹在 `/A` (Action, 类别为 `/GoTo`) 字段里，解析器需要具备兼容这两种标准写法的判定逻辑。
