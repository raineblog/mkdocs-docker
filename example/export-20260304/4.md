下面我按你的两个问题来给一套“能长期维护、能精确到页内坐标（H2 不在页顶也能跳到原位置）”的方案。为了便于讨论，我默认你说的“植入目录”主要指 **PDF 的书签 / Outlines（左侧导航树）**，以及（可选）一份**可见的目录页（TOC pages）**。

## 总结（先给结论）

- **(1) Python 操控 PDF：**  
  如果你能接受 **AGPL**（或愿意买商业授权），我最推荐 **PyMuPDF**：它同时擅长“合并/插页/移动页 + 读取链接目标坐标 + 写书签（TOC/Outlines）”，非常适合你这种要“精确到垂直高度”的需求。它的 `Page.get_links()` 能直接拿到内部链接的目标页和目标点坐标。  
  如果你不能接受 AGPL：就用 **pikepdf（合并/结构修复/稳） + pypdf（写书签/页标签）** 的组合；或者只用 pikepdf 也能写 outlines，但“页内精确定位”的数据来源与坐标提取你还是需要额外手段。

- **(2) XeLaTeX 生成页面的组织方式：**  
  最好是 **XeLaTeX 生成“若干块（chunks）PDF”**（例如：封面块、扉页块、目录块、尾页块、章首页块），然后在 Python 里一次性组装成最终 PDF，**最后再写入书签/目录**。  
  不建议“先在源 PDF 里预留空页再覆盖”的方案：它对页数/插入点变动非常敏感，而且**对你想要的‘跳到 H2 原始垂直位置’几乎没有帮助**（书签定位是页内坐标问题，不是空页问题）。

## (1) Python 操控 PDF：有哪些库、优缺点、我推荐哪个

下面这个表格是按你关心的能力来对比：**合并/插页、保留链接、生成书签（Outlines）、能否拿到页内坐标（y）、许可证与 CI 友好度**。

### 1) PyMuPDF（强烈匹配你的“精确到垂直高度”诉求）

**优点**
- **合并/插页能力强**：`Document.insert_pdf()` 就能拼接，属于它官方教程里最常见的用法之一。  
- **能读到“内部链接”的目标页+目标坐标**：`Page.get_links()` 返回的字典里包含 `page`（0-based）与 `to`（目标点坐标）。这对你要把“网页提取的 h1~h6 目录定位到页内具体高度”非常关键。  
- **能写入书签（TOC/Outlines），且支持“高度/坐标”**：`Document.set_toc()` 支持简单条目，也支持带 `dest` 字典的精细控制；文档里明确提到 `dest` 可以是“高度数值”或 `get_toc(False)` 那种详细字典。并且 v1.23.8 起对 `to` 坐标系还有过一次明确变更说明（这反而说明它在认真处理坐标一致性）。  
- 还能做“可见目录页 TOC 的点击区域链接”之类的事（它官方 recipes 里就展示了：搜索目录文字位置 → 插入 `LINK_GOTO` 链接 → 再 `set_toc()`）。

**缺点**
- **许可证：AGPL / 商业授权**（PyPI 项目说明写得很清楚：开源版本是 AGPL，同时提供商业许可）。如果你的 CI 产物要分发给客户/外部、或你做 SaaS 输出给用户，AGPL 可能会“传染”你的交付方式，需要你评估合规成本。  
- 它功能很多，API 面比 pypdf 大一些，但对你这种需求其实是优点。

**结论**：  
> 只要许可证 OK，PyMuPDF 基本就是为你这类“结构组装 + 精确定位书签”场景量身定做的首选。

---

### 2) pikepdf（更像“PDF 结构工程工具”，稳、修复强、license 友好）

**优点**
- 基于 qpdf，擅长 **修复、线性化、处理各种怪 PDF**，官方文档也明确定位它是低层 PDF 工具，能力覆盖面很强。  
- **支持 Outlines（书签）编辑/创建**，并且可以指定目的地类型与 top 等参数（文档里有例子）。  
- **许可证 MPL-2.0**（PyPI 项目页写明），通常比 AGPL 好用很多：可闭源使用，只要求你对 pikepdf 本身的修改开源（file-based copyleft）。

**缺点（与你的需求强相关）**
- 它擅长“结构层面的 PDF 操作”，但你要做的关键点之一是：**把 h2/h3 精确定位到页内 y 坐标**。  
  pikepdf 自己不负责渲染/版面分析，它当然能把“某个 y”写进目的地，但**这个 y 从哪里来**，你还需要别的办法（比如：  
  - 从 Chrome 打印出来的 PDF 内部链接（GoTo）里提取；或者  
  - 额外用能“读文字块 bbox”的库去算；pikepdf 本身不擅长这个方向）。

**结论**：  
> 如果你不能用 PyMuPDF，pikepdf 是“合并与修复”的首选底座，但你需要配一个“坐标/文本定位”的手段。

---

### 3) pypdf（纯 Python、写书签/页标签很顺手）

**优点**
- 纯 Python：CI 上安装通常省心。  
- 写书签很直接：`add_outline_item()`，而且 Fit/XYZ 这种目的地形式它也支持（文档示例里有 `Fit.fit_horizontally(top=800)`、`Fit.xyz(...)`）。  
- **页标签 Page Labels**：`PdfWriter.set_page_label(...)` 这种把前言改成罗马数字、正文从 1 开始的需求，它直接支持（非常适合“封面 i, ii, iii / 正文 1,2,3”这类出版物习惯）。  
- 许可证 BSD-3-Clause（PyPI 元数据）。

**缺点**
- 对“各种奇怪 PDF 的健壮性/修复能力”一般不如 qpdf/pikepdf。  
- 同样存在关键问题：**你要的页内 y 坐标从哪里来？** pypdf 能写目的地，但不擅长“版面定位/提取 bbox”。

**结论**：  
> pypdf 非常适合作为“写书签/页标签”的工具，但如果你要“自动精准定位到 H2 的原始 y”，你通常还得再配一个“坐标来源”。

---

### 4) 其他库（简单点评）
- **pdfrw**：老牌、能改对象，但生态与“高级导航/书签/坐标”不如上面三者；如果只是 PageLabels，有个小库 `pagelabels` 是基于 pdfrw 的。  
- **borb**：也能做很多事，但授权/版本策略经常让工程落地不如 pypdf/pikepdf 省心（这里不展开）。  
- **reportlab**：强在“生成新 PDF”，弱在“改已有 PDF 结构”；通常用来做盖章、页码 overlay 等。

---

### 我的推荐（按你的真实需求分两档）

#### A 档：你能用 AGPL / 或可以买商业许可  
**就选 PyMuPDF，一把梭。**  
原因：你想要的“精确到 H2 原始垂直高度”，最可靠的数据来源往往是：
- **从 PDF 里读取内部链接的目标点**（GoTo destination），或
- **从 PDF 里定位文字 bbox**，换算出 y；  
这两类都属于 PyMuPDF 的强项（`get_links()` / `search_for()` / `get_text("dict")` 等）。

#### B 档：你不能用 AGPL  
**推荐组合：pikepdf（负责合并/修复/结构稳定） + pypdf（负责写 outlines + 页标签）**。  
但注意：你仍然要解决“y 坐标来源”。工程上通常有两条路：
1) **让 Chrome 打印时生成内部链接**（例如你的网页目录里是 `<a href="#id">` 的锚点链接），PDF 里会有 link annotations，然后用能读取 link destination 的工具提取目标坐标。PyMuPDF 读取最方便，但如果你不能用它，就要找别的能读链接目的地的方案（实现成本会上来）。  
2) **用文本抽取 + bbox 反查**（搜索每个标题字符串出现在哪一页、哪个矩形区域），得到 y。PyMuPDF 最成熟；其他库能做到但要更费劲。

## (2) XeLaTeX 构建页面：分散生成 vs 统一生成 vs 预留空页覆盖，哪个更好？

你提出的三个方案我逐个评价，但会把重点落在你最在意的两点：
1) **维护性 / 灵活插页**  
2) **书签能精确跳到 H2 在源 PDF 中的垂直位置**

### 方案对比结论（先说人话版）
- **“预留空页再覆盖”**：最不推荐。它让系统对页数高度耦合，而且对“页内 y 位置跳转”没本质帮助。  
- **“XeLaTeX 全部生成成一个 PDF，再在 Python 里插入到各处”**：适合“前置固定页面（封面、扉页、版权页、总目录、尾页）”这类结构，维护性很好。  
- **“XeLaTeX 每种页面（或每章）生成多个小 PDF，再在 Python 里拼装”**：适合“章首页/分隔页这种需要按章动态生成”的部分；但不要细到“每一页一个 PDF”（文件管理和编译成本会膨胀）。

下面是更工程化的建议。

---

## 我建议你采用的“最稳的组装策略”（兼顾精确 y 的书签）

### 核心原则 1：**网页打印出的 content PDF 尽量保持页内容不动**
你要“定位到 H2 原来的位置（源 PDF 的垂直高度）”。最稳的方法是：

- 把 Chrome 打印得到的 **正文 content.pdf** 当作“不可变正文块”
- Python 只做：
  - 在页与页之间插入新页（封面、章首页等）
  - 合并 PDF
  - 写 Outlines（书签）
  - （可选）写 Page Labels（前言罗马数字、正文阿拉伯数字）
- **不要对正文页做 reflow/重排**，否则你想要的“源 PDF y 坐标”就不再成立

这样，H2 在某一页内的 y 坐标始终稳定；你插入再多页，也只是“页号整体偏移”，不会破坏“页内高度”。

---

### 核心原则 2：书签的“y 坐标”最好来自 PDF 自己，而不是来自 HTML
你已经意识到：H2 可能不在页顶，你希望书签跳转到原位置。

**最省维护、最不怕 CSS/分页变化的做法**通常是这两种之一：

#### 做法 A（强烈推荐）：利用“PDF 内部链接”作为坐标真源
让网页里生成一个目录（可以隐藏），目录项是 `<a href="#h2-id">xxx</a>` 这种锚点链接。Chrome 打印后，PDF 往往会保留内部链接 annotation；然后你在 Python 里读取这些 links：

- link 的 `page` 就是目标页（0-based）
- link 的 `to` 就是目标点坐标（包含 y）

PyMuPDF 读这个最直接：`Page.get_links()` 明确返回 `page` 与 `to`。  
你随后把这些目标写进最终 PDF 的 Outlines（再加上你插入的封面/章首页造成的页号 offset）即可。

> 这条路最大的价值：**坐标来自最终排版后的 PDF**，不会因为 HTML 的 bbox、缩放、分页策略变化而失真。

#### 做法 B：在 PDF 中“搜索标题文字 → 取 bbox → 得到 y”
如果你不想依赖内部链接，就从 PDF 文本层里找标题出现位置，取它的矩形 bbox（y 就出来了）。PyMuPDF 的 recipes 里就展示了“search_for 得到 rect，再插 link”的套路。  
缺点是：标题重名、断行、字体子集等情况下要做更多容错。

---

## 三种 XeLaTeX 组织方式的建议

### 方式 1：XeLaTeX “每个页面一个 PDF” → Python 合并
**不推荐作为主方案**（除非你页面数量极少）。

- 优点：单页粒度很灵活
- 缺点：文件数量爆炸；编译次数多；你改模板要批量重编；CI 时间会上去
- 对“精确 y 书签”没有额外增益（书签精确与否取决于你怎么拿 y）

### 方式 2：XeLaTeX “生成一个包含所有前置/后置页面的大 PDF” → Python 插入/拼接
**推荐用于“前言/封面/扉页/版权页/总目录/尾页”等固定结构块。**

- 优点：模板集中；编译一次；维护成本低
- 缺点：如果“章首页”是按章动态变化，那这一块不太适合全部塞进去（除非你也把章首页作为同一个文档里多页生成）

### 方式 3：XeLaTeX “生成若干块 PDF（chunks）” → Python 组装
**我认为这是最符合你目标的折中最优解**：

建议分成这些 chunk（举例）：
- `front_fixed.pdf`：封面、扉页、版权页、前言（不含目录）
- `toc.pdf`：可见目录页（如果你要做）
- `chapter_openers/xx.pdf`：每章一页（或两页）章首页（动态）
- `content.pdf`：Chrome 打印正文（尽量不动）
- `back_fixed.pdf`：附录分隔页、尾页、封底等

Python 最终组装：
```
final = front_fixed
     + toc
     + (chapter_opener_1 + content_chunk_1)
     + (chapter_opener_2 + content_chunk_2)
     + ...
     + back_fixed
```

这种方式：
- 维护上：LaTeX 模板清晰，每块职责单一
- 插页上：你想把“章首页”插到任何章节前都很精确（按页插）
- 书签上：最终在 Python 里统一写 Outlines，一次性把 offset 全部算好即可

## 关键点：你要的“精确到 H2 垂直高度”如何在最终 PDF 里实现？

### 用 PyMuPDF 写书签（TOC/Outlines）时的关键能力
PyMuPDF 的 `set_toc` 支持你给每个条目一个 `dest`（高度数值或字典），用于精确控制跳转点。

同时，PyMuPDF 能读取链接目标坐标：`get_links()` 的返回里就有 `to` 点。

所以你可以把流程做成：

1) 先组装最终 PDF（插页都完成）  
2) 再计算每个标题在最终 PDF 中对应的 `(page_index, y)`  
3) 调用 `set_toc()` 写入书签

> 这样你完全不需要在“插页前”去维护书签页码，也不怕插入页导致页号漂移。

---

## 额外加分项：页标签（罗马数字/阿拉伯数字切换）
如果你希望 PDF 阅读器的缩略图页码显示：
- 前言 i, ii, iii…
- 正文 1,2,3…

pypdf 的 `PdfWriter.set_page_label()` 是现成的（0-based 页索引范围）。  
（PyMuPDF 也能读到页标签 `Page.get_label()`，但“写页标签”这块 pypdf 通常更直接。）

## 一个实操落地的“推荐流水线”（不需要你现在就改很多）

### Step 0：Node/Chrome 输出两份东西
- `content.pdf`：正文
- `headings.json`（可选）：从 HTML 提取 h1~h6 的层级与文本（不含坐标也行）

### Step 1：Python 组装页面块
- XeLaTeX 产出 `front_fixed.pdf / back_fixed.pdf / chapter_openers/*.pdf / toc.pdf`
- Python 先把它们与 `content.pdf` 组装成 `assembled.pdf`

### Step 2：Python 从 `content.pdf` 获取“标题 → (页, y)”真值
优先用“内部链接目的地”法（最稳）：  
- 如果你在 HTML 里有 TOC 链接（哪怕隐藏），打印后 PDF 里会有 links  
- PyMuPDF `Page.get_links()` 就能拿到目标页与目标点 `to`，作为 y 来源 

不行再用 search+bbox 法：  
- 逐页 `search_for(title_text)`，取第一个 rect 的 y  
- 再写入 outlines

### Step 3：写入最终 Outlines（书签树）
- 用 `set_toc()` 统一写，确保层级合法（level 只能 +1、或回退）。PyMuPDF 对 TOC 结构规则也比较严格，你要按它规则生成。

## 你原来提到的“预留空页覆盖”的方案，为什么我不建议
你设想的思路是：
- Node 先知道每个 PDF 页数
- 在源 PDF 里预留空页
- Python 再覆盖/替换这些空页

这套在“固定插页点”时能工作，但它的问题在于：
1) **耦合页数**：任何 TOC 页数变动、章首页多一页/少一页都会引发连锁反应。  
2) **并不能解决‘页内 y’定位**：H2 的 y 来自正文页内容本身，你插空页不改变它；真正要解决的是“坐标提取 + 书签目的地写入”。  
3) 覆盖/替换页往往更容易引入“链接丢失/注释丢失/资源重复”之类的隐性问题（尤其你正文 PDF 里可能已经有内部链接）。PyMuPDF 在链接/注释操作上是很明确的一套模型（get_links/insert_link/update_link），但“覆盖页”属于更激进的改动，风险更高。

## 最后给一句“选型一句话”
- **你最在意的是：维护性 + 目录书签能精确跳到 H2 的页内原高度。**  
- 这在工程上基本等价于：**你需要一个既能读出 PDF 里 link destination / 文本 bbox 坐标，又能写 Outlines 的库。**  
- 所以：**PyMuPDF（如果许可证 OK）是最贴合你需求的一站式解；否则就要走 pikepdf+pypdf 并额外补齐坐标来源的复杂度。**
