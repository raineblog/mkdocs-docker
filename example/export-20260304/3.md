# Python PDF 库对比 & 架构方案分析

## 一、Python PDF 操控库对比

### 1. PyMuPDF (fitz) ⭐ 推荐

| 维度 | 说明 |
|------|------|
| **优点** | ① 能**精确提取文本及其坐标位置**（bbox），这对你「定位 H2 垂直高度」的需求至关重要；② 合并/拆分/插入页面 API 简洁；③ 书签(outline/TOC) 读写原生支持，且目录项支持精确到 `(page, y_offset)` 级别；④ 性能极好（C 底层）；⑤ 可叠加/覆盖页面（overlay） |
| **缺点** | ① AGPL 协议（商用需购买商业许可）；② 安装包较大 |

**关键能力示例：**
```python
import fitz  # PyMuPDF

doc = fitz.open("source.pdf")
page = doc[2]

# 精确搜索 H2 文本，返回其在页面上的矩形位置（含垂直坐标）
rects = page.search_for("某个 H2 标题")
if rects:
    y_position = rects[0].y0  # 精确的垂直像素位置

# 设置带精确垂直定位的目录
# [level, title, page_number, {"kind": 1, "to": fitz.Point(0, y_position)}]
toc = [
    [1, "第一章", 1, {"kind": 1, "to": fitz.Point(0, 0)}],
    [2, "1.1 节", 3, {"kind": 1, "to": fitz.Point(0, 350.5)}],  # 精确到 y=350.5
]
doc.set_toc(toc)
```

### 2. pikepdf

| 维度 | 说明 |
|------|------|
| **优点** | ① 基于 QPDF，对 PDF 底层结构操控能力极强；② 合并/拆分可靠；③ MPL-2.0 协议友好；④ 能读写 PDF outline（书签）；⑤ 保真度高，不会破坏原 PDF 结构 |
| **缺点** | ① **无法提取文本坐标**——这是致命缺陷，你无法用它定位 H2 的垂直位置；② 书签的精确 y 定位需要你手动构造底层 PDF 字典对象，API 较低层 |

```python
# pikepdf 写书签需要手动构建 PDF 对象
from pikepdf import Pdf, OutlineItem, make_page_destination
# 能做，但写 /XYZ 0 y 0 这种精确目标需要较多底层代码
```

---

### 3. pypdf (原 PyPDF2)

| 维度 | 说明 |
|------|------|
| **优点** | ① 纯 Python，零外部依赖；② 合并/拆分 API 简单；③ BSD 协议；④ 社区活跃 |
| **缺点** | ① **无法提取文本坐标**；② 书签写入支持有限，精确 y 定位需要 hack；③ 性能一般；④ 对复杂 PDF 兼容性偶有问题 |

---

### 4. ReportLab

| 维度 | 说明 |
|------|------|
| **优点** | ① PDF 生成能力极强（从零构建 PDF）；② 精确排版控制 |
| **缺点** | ① 主要用于**生成** PDF，不擅长**操控**已有 PDF；② 无法读取/解析已有 PDF 的内容和坐标；③ 对你的场景（合并已有 PDF）几乎无用 |

---

### 5. pdfrw / borb

| 维度 | 说明 |
|------|------|
| **pdfrw** | 能合并和操控 PDF 结构，但已不太活跃，且无文本坐标提取 |
| **borb** | API 较现代，但生态小、文档少、稳定性存疑 |

---

### 📊 综合对比矩阵

| 能力 | PyMuPDF | pikepdf | pypdf | ReportLab |
|------|---------|---------|-------|-----------|
| 合并/拆分/插入页面 | ✅ 优 | ✅ 优 | ✅ 良 | ❌ |
| 提取文本**坐标** | ✅ **独有优势** | ❌ | ❌ | ❌ |
| 书签读写(含 y 精确定位) | ✅ 简洁 API | ✅ 底层可控 | ⚠️ 有限 | ❌ |
| 页面叠加/覆盖 | ✅ | ✅ | ⚠️ | ❌ |
| PDF 保真度 | ✅ | ✅ 最佳 | ⚠️ | — |
| 协议 | AGPL | MPL-2.0 | BSD | BSD/商业 |
| 性能 | 🚀 最快 | 🚀 快 | 🐢 慢 | — |

### 🏆 结论

> **首选 PyMuPDF**——它是唯一能同时满足「提取文本精确垂直坐标 + 写入精确 y 定位书签 + 合并插入页面 + 页面覆盖叠加」的库。你的核心需求"定位到 H2 原来的位置"只有它能直接做到。
>
> 如果 AGPL 协议有顾虑，可用 **pikepdf + PyMuPDF 组合**：用 PyMuPDF 只做文本坐标提取（读取阶段），用 pikepdf 做最终的合并/书签写入（写入阶段），这样可以把 AGPL 的影响限制在最小范围。

## 二、架构方案对比

你提出了三种方案，我逐一分析后给出推荐：

### 方案 A：XeLaTeX 逐页构建 → Python 合并

```
XeLaTeX → cover.pdf, title.pdf, toc.pdf, chapter1_title.pdf, ...
Node    → content1.pdf, content2.pdf, ...
Python  → 按顺序合并所有 PDF + 植入目录
```

| 优点 | 缺点 |
|------|------|
| 每个页面独立，最灵活 | PDF 文件数量多，管理复杂 |
| 修改单页不影响其他 | 目录页需要知道最终页码，存在**循环依赖** |
| 合并顺序完全由 Python 控制 | — |

### 方案 B：XeLaTeX 构建所有特殊页合一 → Python 插入内容页

```
XeLaTeX → special_pages.pdf（含封面/扉页/目录/章首/尾页，中间留位置）
Node    → content1.pdf, content2.pdf, ...
Python  → 把内容页插入到 special_pages.pdf 的相应位置
```

| 优点 | 缺点 |
|------|------|
| XeLaTeX 内部可以自己管理特殊页的排版 | 插入位置计算复杂，章节页数变化会导致位置偏移 |
| 文件少 | 不够灵活，修改一处可能影响全部 |

### 方案 C：XeLaTeX 构建含空白占位页 → Python 覆盖替换

```
XeLaTeX → full_book.pdf（特殊页正常，内容页为空白占位）
Node    → content1.pdf (3页), content2.pdf (5页), ...  # Node 输出页数
Python  → 用内容页覆盖对应的空白页
```

| 优点 | 缺点 |
|------|------|
| XeLaTeX 可以直接算总页码和目录页码 | **致命问题**：空白页数量必须和内容页完全匹配 |
| — | 如果内容变化，空白页数量需要同步更新，**脆弱** |
| — | 覆盖操作不如插入直观 |

---

### 🏆 推荐：方案 A 的改进版（两轮构建）

方案 A 最灵活，但有一个「目录页码循环依赖」的问题。解决方案是 **两轮构建**：

```
┌─────────────────────────────────────────────────────────┐
│                    完整流水线                              │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  Step 1: Node + Chrome 打印网页 → content_*.pdf          │
│          同时输出 JSON 元数据:                             │
│          {                                               │
│            "file": "content_ch1.pdf",                    │
│            "pages": 12,                                  │
│            "headings": [                                 │
│              {"level": 1, "text": "章标题", "page": 0,   │
│               "y": 0},                                   │
│              {"level": 2, "text": "节标题", "page": 2,   │
│               "y": 345.6},                               │
│              {"level": 3, "text": "小节",   "page": 2,   │
│               "y": 720.1}                                │
│            ]                                             │
│          }                                               │
│                                                          │
│  Step 2: Python 第一轮 — 计算页码布局                      │
│          已知: 封面1页 + 扉页1页 + 目录?页 + 各章内容页数    │
│          估算目录页数 → 确定所有最终页码                     │
│                                                          │
│  Step 3: XeLaTeX 构建各特殊页（传入页码参数）               │
│          xelatex '\def\tocpagestart{5}                   │
│                   \def\chOneStart{8}                     │
│                   ... \input{toc.tex}'                   │
│          → cover.pdf, title.pdf, toc.pdf,                │
│            chapter1_title.pdf, ..., back.pdf             │
│                                                          │
│  Step 4: Python 第二轮 — 最终合并                          │
│          cover + title + toc +                           │
│          chapter1_title + content_ch1 +                  │
│          chapter2_title + content_ch2 +                  │
│          ... + back                                      │
│                                                          │
│  Step 5: Python 植入 PDF Outline (书签/目录)              │
│          用 PyMuPDF 精确到 y 坐标                         │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 关于「精确到 H2 垂直高度」的实现

这是你需求中最关键的部分。有 **两条路径** 获取 y 坐标：

#### 路径 1：Node 端直接输出坐标（推荐 ✅）

```javascript
// 在 Chrome/Puppeteer 中，打印前提取所有标题的位置
const headings = await page.evaluate(() => {
    return [...document.querySelectorAll('h1,h2,h3,h4,h5,h6')].map(h => {
        const rect = h.getBoundingClientRect();
        return {
            level: parseInt(h.tagName[1]),
            text: h.textContent.trim(),
            // 关键：计算在哪一页、页内 y 坐标
            // 假设 A4: 页高 = 297mm ≈ 841.89pt
            pageIndex: Math.floor(rect.top / PAGE_HEIGHT_PX),
            yInPage: rect.top % PAGE_HEIGHT_PX,
            // 转换为 PDF 点数（1px ≈ 0.75pt，但取决于 DPI 设置）
        };
    });
});

// 输出到 JSON，供 Python 消费
fs.writeFileSync('ch1_meta.json', JSON.stringify(headings));
```

> **优点**：坐标来自渲染引擎本身，最精确，且和打印 PDF 完全一致。

#### 路径 2：Python 端用 PyMuPDF 提取（备选）

```python
import fitz

doc = fitz.open("content_ch1.pdf")
headings = []

for page_idx, page in enumerate(doc):
    # 提取带格式的文本块
    blocks = page.get_text("dict")["blocks"]
    for block in blocks:
        for line in block.get("lines", []):
            for span in line["spans"]:
                # 通过字体大小/粗细判断是否为标题
                if span["size"] > 18 and "Bold" in span["font"]:
                    headings.append({
                        "level": guess_level(span["size"]),
                        "text": span["text"],
                        "page": page_idx,
                        "y": span["bbox"][1],  # 精确的 y 坐标（从页面顶部算起）
                    })
```

> **缺点**：需要靠字体大小/名称反推标题层级，不如 Node 端直接从 DOM 获取准确。

#### 推荐组合

```
Node 端 → 输出 headings.json（含 level, text, page, y）
Python 端 → 直接读 JSON，无需再猜
```

### 最终植入 PDF 书签的代码骨架

```python
import fitz
import json

def build_final_pdf(config):
    # 1. 按顺序合并所有 PDF
    final = fitz.open()

    page_offset = 0
    structure = []  # [(name, start_page, num_pages), ...]

    for part in config["parts"]:
        # part: {"type": "cover"|"toc"|"chapter_title"|"content"|"back",
        #        "file": "xxx.pdf", "meta": "xxx.json"(optional)}
        src = fitz.open(part["file"])
        final.insert_pdf(src)
        structure.append((part["type"], page_offset, len(src)))
        page_offset += len(src)

    # 2. 构建目录书签（TOC / Outline）
    toc = []

    for part in config["parts"]:
        if part["type"] == "content" and "meta" in part:
            meta = json.load(open(part["meta"]))
            content_start_page = get_start_page(structure, part)

            for h in meta["headings"]:
                absolute_page = content_start_page + h["page"]
                # PDF 坐标系 y 轴从底部向上，PyMuPDF 已处理为从顶部向下
                # kind=1 表示 /XYZ 类型目标（含精确坐标）
                toc.append([
                    h["level"],
                    h["text"],
                    absolute_page + 1,  # PyMuPDF TOC 页码从 1 开始
                    {
                        "kind": 1,        # /XYZ
                        "to": fitz.Point(0, h["y"]),  # 精确垂直位置！
                        "zoom": 0,        # 保持当前缩放
                    }
                ])

    # 3. 写入书签
    final.set_toc(toc)

    # 4. 保存
    final.save("output.pdf", garbage=4, deflate=True)

build_final_pdf(config)
```

### 📋 总结

| 决策点 | 推荐 |
|--------|------|
| Python PDF 库 | **PyMuPDF**（精确坐标 + 书签 + 合并一站式） |
| 架构方案 | **方案 A 改进版**（逐个构建 + 两轮计算页码） |
| 标题坐标获取 | **Node 端直接输出 JSON**（最精确，避免反推） |
| 目录页码 | Python 先算布局 → 传参给 XeLaTeX → 再最终合并 |
| 书签精度 | `fitz.Point(0, y)` 精确到 PDF 点数级别 |

这套架构的好处是**每一层职责清晰**：Node 负责网页→PDF + 元数据，XeLaTeX 负责排版特殊页，Python 负责编排和组装——任何一层变化都不影响其他层。
